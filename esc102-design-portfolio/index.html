Project Map

Projects featured here

Praxis I

Backpack Strap Design

Team-based redesign work focused on excess strap snagging, adjustability, comfort, and manufacturability.

CTMFs carried by this case

  • Design concepts Three concept directions stayed distinct long enough to expose genuinely different tradeoffs.
  • Evaluation criteria Snagging, adjustability, comfort, and manufacturability became explicit criteria instead of afterthoughts.
  • Proxy testing A simplified simulation and submitted evidence acted as low-resource stand-ins for full ergonomic testing.

CIV102

Bridge Design

Matboard bridge design supported by drawings, calculations, and analysis code for load envelopes and failure modes.

CTMFs carried by this case

  • Requirements Span, load, material rules, and fabrication limits all acted as active design requirements.
  • Verification Calculations, envelopes, and drawings were used to check whether the bridge logic survived contact with build reality.
  • Prototypes: graphical The drawing package worked as a graphical prototype for spacing, glue logic, and assembly.

Praxis II

Daycare Wristband Workflow

Team-based workflow redesign that converged on a parent-led two-station system with a loading-aid prototype.

CTMFs carried by this case

  • Evaluation criteria The requirement table converted workload, safety, intuitiveness, and setup concerns into explicit metrics.
  • Workflow prototype The chosen direction became clearer once the device and the two-station parent flow were read together.
  • Sensitivity analysis Validation results showed where the chosen direction stayed useful under changing assumptions.

Independent Study

Gaussian Entanglement Project

Analytic and computational work on entanglement thresholds in symmetric phase-insensitive Gaussian channels.

CTMFs carried by this case

  • Analytical The central move was turning the entanglement condition into an analytical claim that could be read in channel parameters.
  • Toulmin’s structure of argument The derivation, figures, and code had to keep the claim, grounds, and warrant visible together.
  • PIAA The plots only mattered once they were used to perceive, interpret, assess, and act on weaknesses in the claim.

Why These Cases

Different project types, different pressure points

Each case shows a different pressure point in my design process.

Praxis I

Concept pressure

Praxis shows how I frame a messy user problem and how long I can keep options open before that exploration starts working against convergence.

CIV102

Build pressure

The bridge tests whether analysis survives contact with drawings, fabrication tolerances, glue strategy, and actual assembly constraints.

Praxis II

Workflow pressure

Praxis II tests whether a concept that saves staff effort still holds up once timing risk, fallback work, and user compliance are made explicit.

Gaussian

Abstraction pressure

The Gaussian project tests whether simplification earns its authority or only looks convincing because the representation is mathematically clean.
