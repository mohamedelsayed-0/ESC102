Praxis I Concepts

Backpack strap project

These notes show how the project moved from a vague complaint about loose straps to a structured comparison between concept directions.

Concept

Framing

Used in Praxis I

Frame

Framing was the move that kept the project from turning into a neat strap clip fix. Once the problem widened, the later model had an actual decision to support.

What the concept meant here

Framing meant deciding what counted as the problem before treating any concept as a solution. In this project that meant refusing to treat the loose strap as an isolated nuisance.

How I applied it

I helped widen the discussion from snagging alone to a linked set of concerns: adjustability, comfort, access to the bag, and manufacturability. That wider frame then informed the criteria and variables used in the comparison model.

Evidence to inspect

The clearest trace is the shift from requirement notes to explicit simulation inputs and prototype-specific metrics.

Project section Evidence sheet PDF MATLAB model

What it helped reveal

The wider frame made it obvious that a concept could reduce snagging while making access or comfort worse. That stopped the project from rewarding the cleanest-looking fix instead of the best overall direction.

Limitation

The frame still leaned toward what could be parameterized. Physical handling, quick-use behavior, and build effort were present, but not represented as strongly as the modeled quantities.

How I would improve it

I would revisit the frame explicitly after the first round of concept comparisons and add a small physical-use test sooner so the frame is not shaped mostly by what the model can express.

Concept

Prototyping

Used in Praxis I

Represent

Prototyping mattered because it made the concept directions inspectable. Without that move, the team would have been comparing narratives instead of design proposals.

What the concept meant here

Here prototyping was less about polished physical build quality and more about keeping an idea concrete enough to be questioned. A rough representation was still better than a loose verbal claim.

How I applied it

I supported the team in preserving three distinct directions, then built a simplified comparison model so each direction could be expressed through parameters such as excess strap, load distance, and pressure rather than just intuition.

Evidence to inspect

The submission page and MATLAB file show the concepts being carried into metrics rather than left as sketches with no test structure.

Project section Submitted PDF MATLAB model

What it helped reveal

Prototyping turned concept comparison into something the team could actually argue with. It exposed differences in load path and obstruction that were not visible when the concepts were only described verbally.

Limitation

The project relied more on representational prototyping than on quick physical checks. A model can clarify structure, but it can also hide how incomplete the supporting evidence still is.

How I would improve it

I would pair the early model with faster physical rig tests or mockups. That would keep the representation useful while making sure the prototype still answers to real use conditions.

Concept

Diverging

Used in Praxis I

Diverge

Diverging was useful because it protected the differences between concepts. It also exposed one of my weak points: I can keep options open longer than the schedule can afford.

What the concept meant here

Diverging did not mean generating as many ideas as possible. It meant holding several plausible directions open long enough to learn what each one solved differently.

How I applied it

I helped keep the elastic, inside-bag adjustment, and shoulder-side adjustment concepts distinct while the team clarified comparison criteria. That preserved meaningful differences instead of collapsing everything into an early hybrid.

Evidence to inspect

The evidence sheet and project summary show three separable concept directions being compared against explicit metrics rather than one merged answer from the start.

Project section Evidence sheet PDF

What it helped reveal

Divergence made it possible to see that the concepts did not fail for the same reason. One direction handled obstruction better, another protected adjustability, and another raised stronger manufacturability questions.

Limitation

In this project divergence lasted slightly too long. By the time the concepts were well understood, the remaining convergence time was tighter than it should have been.

How I would improve it

I would set an exit condition before diverging begins: what evidence is needed, what distinction each concept is supposed to reveal, and when the team is required to start closing options.

CIV102 Concepts

Bridge project

These entries show where the bridge stopped being a clean analytical exercise and became a design problem tied to geometry, assembly, and build quality.

Concept

Failure-mode framing

Used in CIV102

Frame

Failure-mode framing made the bridge legible. Instead of asking whether the bridge was “strong,” the design work became a question of where and how it was most likely to fail.

What the concept meant here

Framing meant decomposing the bridge into tension, compression, buckling, shear, glue, and local stability concerns. That decomposition created a map of design pressure rather than a single pass-fail target.

How I applied it

I used calculations and code to treat different bridge regions and different failure types explicitly. That kept the team from reading one strong number as proof that the whole design was well resolved.

Evidence to inspect

The calculations and analysis script both separate capacities and safety factors by mode and location instead of collapsing the bridge into one metric.

Project section Calculation package Analysis code

What it helped reveal

This frame showed that several geometry choices were not making the bridge universally better; they were shifting weakness from one mode to another. That made splice placement, diaphragm spacing, and flange decisions much easier to read.

Limitation

The frame still underrepresented fabrication quality. Glue application, cutting precision, and assembly consistency were acknowledged, but they were not modeled with the same explicitness as the structural modes.

How I would improve it

I would add a parallel constructability frame at the start: expected tolerance risk, glue-critical joints, and steps in the build sequence that could invalidate an otherwise strong analytical design.

Concept

Tradeoff iteration

Used in CIV102

Converge

Iteration was the convergence tool in this project. The bridge only improved when geometry was treated as revisable under several competing effects instead of optimized once against a single clean criterion.

What the concept meant here

Tradeoff iteration meant revising the bridge while keeping several capacities visible at once: bending, shear, buckling, stiffness, glue demand, and the practical cost of the section shape.

How I applied it

I used section-property work and load-envelope reasoning to compare geometry choices and track how each revision improved one condition while often making another condition tighter.

Evidence to inspect

The design calculations and envelope-analysis script show repeated recalculation around changing geometry rather than a one-pass optimization.

Project section Calculation package Analysis code

What it helped reveal

Iteration exposed how quickly an efficient-looking section could become risky once buckling resistance, glue lines, or local stability were brought back into view. It helped convergence resist false neatness.

Limitation

Most of the iteration still happened numerically before the drawings fully pushed back. That meant some revisions were analytically justified earlier than they were buildably justified.

How I would improve it

I would require a buildability check at each major revision. If a geometry change cannot survive a drawing or assembly review, it should not count as a genuine design improvement.

Concept

Representing

Used in CIV102

Represent

Representing mattered because the bridge was never complete as an analysis alone. The drawings were the point where geometry, spacing, and assembly assumptions became visible enough to challenge.

What the concept meant here

Representing meant treating drawings as part of the design process rather than as downstream documentation. A drawing was useful only if it could expose hidden assumptions in the analysis.

How I applied it

I used the drawing package as a way to test whether the analytical bridge could actually be cut, aligned, glued, and assembled with the relationships the calculations assumed.

Evidence to inspect

The engineering drawings and design report show the analytical design translated into dimensions, spacing logic, and assembly decisions that other teammates could act on.

Project section Engineering drawings Design report

What it helped reveal

Representation exposed where the bridge was relying on idealized geometry. Once the design had to be drawn for actual construction, tolerance, sequence, and accessibility stopped being background issues.

Limitation

Drawings can still look resolved while hiding how difficult assembly will be in practice. Clean lines on a page are not yet proof that fabrication friction has been accounted for.

How I would improve it

I would bring drawings into the project earlier and annotate them with tolerance concerns, glue sequence, and handoff points so representation starts shaping the design before convergence hardens.

Gaussian Concepts

Independent analytic work

These notes focus on method design: what abstraction bought, where it risked overclaiming, and how benchmarking kept the cleaner result accountable.

Concept

Reduction

Used in Gaussian project

Frame

Reduction was the key framing move in this project. It turned a repeated numerical condition into something interpretable at the parameter level, which is where the project started producing design leverage.

What the concept meant here

Reduction meant redesigning the problem under a symmetric setting so the entanglement threshold could be written directly in transmissivity, noise, gain, and squeezing parameters.

How I applied it

I reframed the partially transposed eigenvalue condition into an analytically cleaner form and then organized the rest of the work around what that new representation made visible.

Evidence to inspect

The long note and benchmark code both center the move from full-spectrum evaluation to a reduced threshold expression.

Project section Long note PDF Benchmark script

What it helped reveal

The reduction created direct threshold statements instead of leaving the project at the level of repeated computation. That made regime comparison, plotting, and later benchmarking much more informative.

Limitation

The leverage comes from symmetry, which also means the method excludes broader channel settings. If that boundary is not stated explicitly, the cleaner result can start looking more universal than it is.

How I would improve it

I would state the validity boundary more aggressively, ideally with a short map of which asymmetric or more general cases are outside the reduction and why.

Concept

Sweeps

Used in Gaussian project

Diverge

Sweeps were the divergence move after the reduction was in place. They kept one elegant example from doing the argumentative work of an entire parameter regime.

What the concept meant here

Sweeps meant exploring parameter space systematically enough that variation itself became evidence. In this project divergence happened across regimes, not across physical product concepts.

How I applied it

I explored transmissivity, thermal occupancy, gain, and squeezing across pure-loss, thermal-loss, and amplifier settings to see where the reduced condition preserved the relevant structure and where slack accumulated.

Evidence to inspect

The heatmaps and regime plots come directly from the sweep logic in the benchmark script.

Project section Benchmark script

What it helped reveal

The sweeps showed where the shortcut stayed informative and where it drifted. That made approximation error part of the design argument instead of an afterthought hidden behind one carefully chosen figure.

Limitation

Sweeps can still be curated to flatter the result. Range choice, resolution, and which regimes are included all shape what the reader is allowed to notice.

How I would improve it

I would justify the selected ranges more explicitly and add a short explanation of which regimes are physically or mathematically important rather than letting figure aesthetics decide the sweep.

Concept

Benchmarking

Used in Gaussian project

Converge

Benchmarking is what kept the reduction honest. It forced the cleaner threshold to answer to exact quantities before I treated it as a dependable guide.

What the concept meant here

Benchmarking meant comparing the reduced threshold and associated bound against exact log-negativity behavior instead of accepting analytic elegance as proof of usefulness.

How I applied it

I generated comparison plots in representative regimes and used them to inspect where the simplified result tracked the trend, where it introduced slack, and where it stopped being a reliable stand-in.

Evidence to inspect

The pure-loss and thermal-loss plots, together with the script that generated them, show exact-versus-bound comparison directly.

Project section Benchmark script Long note PDF

What it helped reveal

Benchmarking showed that the reduction was not simply “right” or “wrong.” It exposed where the shortcut preserved the important structure and where the gap between exact and reduced behavior became large enough to matter.

Limitation

The benchmark set is still a curated sample. Even good comparison plots can create confidence too quickly if they are not paired with a clear statement about untested regimes.

How I would improve it

I would add a compact error summary across regimes and intentionally include a few harder edge cases, so the benchmark does more than confirm the method where it already looks strong.
